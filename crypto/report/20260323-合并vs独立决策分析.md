# My-Stock 与加密货币量化交易：合并 vs 独立决策分析
**创建时间**: 20260323 22:30

## 结论：合并到 my-stock

**推荐在 my-stock 中新增 `app/crypto/` 模块，而非独立建项目。**

---

## 一、决策依据

### 1.1 已有基础盘点

| 资产 | 状态 | 位置 |
|------|------|------|
| 架构分析报告 | 已完成 | `arc/report/2026-03-17-crypto-quant-architecture-analysis.md` |
| 加密货币研究框架 | 已搭建 | `crypto/scripts/`（10个模块，含数据获取/策略/回测/交易） |
| 数据库设计 | 已完成 | `stock_crypto` 库，4张核心表 |
| OKX 实操指南 | 已完成 | `crypto/report/20260323-OE量化交易实操指南.md` |
| my-stock 加密代码 | **无** | `F:\projects\my-stock\app\` 下无 crypto 相关代码 |

### 1.2 可复用组件（来自架构分析报告）

**直接复用（不改或极少改）：**
- 数据库连接层 `database.py` + `db_utils.py`（batch_upsert）
- 三层缓存 `UnifiedCache`
- 优先队列 `AkShareQueue` / `MultiSourceQueue`
- 调度器 `scheduler_all.py`（JSON注册表驱动）
- 策略接口 `BaseStrategy`（DataFrame输入/信号输出，市场无关）
- 风控 `risk_manager.py`
- 前端：K线图 / 资产曲线 / 认证流程 / Pinia stores
- 日志 / 统一响应 / JWT认证

**需适配：**
- 回测引擎：去印花税+涨跌停 → maker/taker费率
- 模拟交易：去15:30定时评估 → 24/7
- 前端分钟图：9:30-15:00 → 24h滚动窗口
- 配色：A股红涨绿跌 → 国际绿涨红跌

**不可复用（A股专有）：**
- 119个 Tushare 模型、同步服务、注册表
- LA选股模型、宏观经济模型
- AkShare 服务

### 1.3 为什么合并优于独立

| 考量 | 合并（推荐） | 独立 |
|------|------------|------|
| **基础设施** | 零成本复用（~2000行代码） | 需要重写或复制 |
| **维护成本** | 一套系统，改一处全生效 | 两套系统，双倍维护 |
| **跨市场分析** | 天然支持（BTC与A股相关性、对冲策略） | 需要额外集成 |
| **前端展示** | 复用K线图/资产曲线，加个tab | 从零搭建前端 |
| **24/7运行** | my-stock本就是常驻服务（FastAPI长运行） | 同样需要常驻 |
| **开发速度** | 快（脚手架已有） | 慢（需搭建全套） |
| **数据隔离** | `stock_crypto` 独立库，不与 `my_stock` 混 | 同样独立库 |
| **代码耦合** | `app/crypto/` 作为独立子模块，耦合可控 | 无耦合 |

**关键判断**：个人量化系统不需要微服务架构。在代码量和团队规模都很小的情况下，**一个well-organized的单体项目 > 两个重复的小项目**。

### 1.4 什么时候该分离

后续如果出现以下情况再考虑分离：
- 加密货币代码量超过A股部分
- 需要靠近交易所部署（低延迟VPS）
- 需要高频做市/套利（当前定位是信号策略，不需要）

### 2.2 分阶段实施

| Phase | 内容 | 预计时间 | 前提 |

### 2.3 技术要点

**连接层**：CCXT（必选）
```python
# 100+交易所统一API
exchange = ccxt.okx({
    'apiKey': os.getenv('OKX_API_KEY'),
    'secret': os.getenv('OKX_API_SECRET'),
    'password': os.getenv('OKX_PASSPHRASE'),
    'enableRateLimit': True,
})
```

**数据管道**：复用现有模式
```
CCXT fetch_ohlcv() → CcxtConnector(限速) → batch_upsert → MySQL(stock_crypto)
                                              ↑ 复用               ↑ 独立库
```

**策略框架**：复用 BaseStrategy
```python
class CryptoMacdStrategy(BaseStrategy):
    """加密货币MACD策略 —— 复用完全相同的接口"""
    def init_indicators(self, df): ...
    def generate_signals(self, df): ...
```

**费率模型**：回测引擎参数化
```python
# A股
cost_model = StockCostModel(commission=0.0003, stamp_tax=0.001)
# 加密货币
cost_model = CryptoCostModel(maker_fee=0.0008, taker_fee=0.001, funding_rate=True)
```

### 2.4 关键注意事项

1. **API密钥安全** — 环境变量管理，不入库不入代码
2. **网络代理** — 境内访问 Binance/OKX 需走代理 `http://localhost:7890`
3. **数据质量** — 加密货币历史数据有缺失K线和价格毛刺，同步时需清洗
4. **精度处理** — 每个交易对精度不同，必须用 `market['precision']`
5. **滑点差异** — 不能套用A股的滑点假设，加密货币波动大得多

当前 `my-stock-research/crypto/scripts/` 已有完整的研究代码。迁移到 my-stock 的路径：

| 研究代码 | → | 生产代码（my-stock） |
|---------|---|---------------------|
| `exchange_client.py` | → | `app/crypto/services/ccxt_connector.py` |

| `models.py` | → | `app/crypto/models/` (拆分) |

研究阶段的代码保留在 research 项目中继续做策略验证，验证通过的策略再迁移到生产。

## 四、总结

**核心判断**：A股和加密货币量化在数据模型（OHLCV）、策略逻辑（BaseStrategy）、回测框架、前端展示上**高度同构**。差异仅在连接层（交易所API vs 数据商API）和运行模式（24/7 vs 工作日），这些差异通过新增 `app/crypto/` 模块即可解决。

**行动建议**：
1. 在 my-stock 中新增 `app/crypto/` 模块，从 Phase 1（数据层）开始
2. 研究阶段的策略验证继续在 my-stock-research 的 `crypto/` 目录进行
3. 验证通过的策略迁移到 my-stock 生产环境
4. 保持数据库隔离（`stock_crypto` 独立库）
