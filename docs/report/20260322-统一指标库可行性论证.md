# 统一指标库可行性论证报告
**创建时间**: 20260322 16:00

## 一、问题背景

目前 my-stock-research 项目中，各研究主题（MACD/MA/RSI/换手率/波动率）各自独立计算指标，存在以下现状：

| 现状 | 数据 |
|------|------|
| 研究库数量 | 5 个（stock_research/stock_ma/stock_rsi/stock_turnover/stock_volatility） |
| 总表数 | 81 张 |
| 总数据量 | **46.6 GB** |
| MACD 日线 × 3 复权 | 4300 万行，4.3 GB |
| MA 日线 × 3 复权 | 4400 万行，5.8 GB（含 29M 行信号表） |
| RSI 日线 × 4 组 | 更多（含无复权+三复权） |

核心疑问：
1. 每个主题都重复存储 OHLCV 行情 + 自己的指标，能否合并？
2. 前复权/后复权/不复权的使用边界是什么？
3. Tushare `stk_factor_pro` 已有 260 列指标，能复用到什么程度？

## 二、复权问题的本质分析

### 2.1 三种复权模式定位

| 复权模式 | 价格基准 | 适合场景 | 致命缺陷 |
|---------|---------|---------|---------|
| **不复权 (bfq)** | 当日实际成交价 | 真实交易价格、回看历史真实K线 | 送股/拆股日产生虚假跳变，指标计算严重失真 |
| **前复权 (qfq)** | 以最新价为锚，历史价格向下调 | **技术指标计算（趋势/形态/信号）** | 每次复权因子更新，全部历史值都会变 |
| **后复权 (hfq)** | 以上市首日为锚，价格只增不减 | **收益率计算、长期回报统计** | 绝对价格无实际意义，远高于真实价格 |

### 2.2 你的判断是对的

- **前复权适合算指标和趋势**：正确。qfq 保证了价格序列的连续性，MA/MACD/RSI/BOLL 等趋势指标在 qfq 下结果最接近实际交易感受。
- **后复权适合算收益率**：正确。hfq 价格序列单调（不会因新的分红而修改历史），`(hfq_close_T2 / hfq_close_T1 - 1)` 是包含分红的真实总收益率。
- **不复权不适合算指标**：正确。项目约束已明确禁止 bfq 计算个股指标。

### 2.3 关键结论：qfq 的"不稳定性"其实不影响指标库

前复权的"历史值会变"听起来可怕，但实际影响很小：

1. **RSI/MACD 等比率指标**：基于 close.diff() 或 close 的比率计算，qfq 和 hfq 的结果**数学上完全等价**（已被研究验证，差异 < 2%）
2. **MA/BOLL 等绝对价格指标**：qfq 下绝对值会随复权更新而变，但相对位置关系（金叉/死叉/突破/乖离率）不变
3. **除权事件频率低**：A 股年均除权 1-2 次，只有除权日当天需要刷新历史数据
4. **Tushare stk_factor_pro 已经帮你做了**：它每天提供 bfq/qfq/hfq 三种复权的指标值，可以直接用

**结论：前复权 qfq 是统一指标库的最佳选择，用于技术指标计算和信号检测。收益率计算单独用 hfq 价格或 adj_factor 转换。**

## 四、可行性论证：能否建立统一指标库？

### 4.1 方案对比

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A. 维持现状** | 各主题独立计算+存储 | 灵活，参数可控 | 46GB 重复存储，OHLCV 冗余 4-5 次 |
| **B. 直接用 stk_factor_pro** | 只用 Tushare 已算的 | 零计算成本，数据权威 | 只有日线，缺 RSI14，无信号，参数不可控 |

## 五、前复权"不稳定性"问题的解决方案

你担心的核心问题：qfq 历史值会因除权而全部改变，存下来的指标岂不是"过期"了？

### 5.1 为什么不是大问题

1. **stk_factor_pro 每天都是最新复权值**：Tushare 每天重新计算，你同步的就是当天最新的 qfq 指标值
2. **研究关注的是相对关系，不是绝对值**：RSI=30 就是 30，MACD 金叉就是金叉，不会因为复权更新而改变
3. **除权当天自动更新**：同步脚本跑完，当天所有值都是最新的

### 5.2 真正需要关心的场景

| 场景 | 影响 | 解法 |
|------|------|------|
| 回看 3 年前某天的 MA20 绝对值 | qfq 下当时的值和现在查到的不同 | 这是正常的，技术分析关注的是当时的相对位置 |
| 计算历史信号的收益率 | 用 qfq 价格算出的收益率会有偏差 | **用 hfq 或 adj_factor 算收益率，不用 qfq** |
| 增量同步 vs 全量重算 | 日线只需增量同步当天 | 周线/月线在除权日需重算最近 N 周期 |

### 5.3 最佳实践

```
指标计算 → qfq（趋势准确）
收益率计算 → hfq 或 adj_factor（绝对值准确）
信号回溯 → 指标用 qfq，收益用 hfq，两者分开
```

## 七、实施建议

### 7.1 如果决定做统一指标库

**Phase 1：搭建统一基础层（1-2 天）**
1. 创建 `stock_indicator` 数据库
2. 建 `unified_daily_qfq` 表，从 stk_factor_pro ETL 同步（每日增量）
3. 补算 RSI14 和 BIAS(5/10/20/60)
4. 建 `hfq_return` 预算表（adj_factor + market_daily → 5/10/20/60 日收益率）

**Phase 2：周线/月线计算（1 天）**
1. 从日线 OHLCV 聚合出周线/月线 K 线
2. 在聚合后的 K 线上计算全部指标
3. 复用现有 batch compute 模式

**Phase 3：信号层迁移（2-3 天）**
1. 修改各主题信号检测脚本的数据源：从各自指标表 → 统一指标库
2. 信号表结构不变，只是读取来源变了
3. 验证信号结果一致性

**Phase 4：清理旧表（可选）**
1. 确认迁移后信号结果一致
2. 旧的指标数据表（stock_macd_daily_xxx 等）可归档或删除
3. 预计释放 ~30 GB 存储

### 7.2 如果不做统一，维持现状的优化建议

1. **至少砍掉多复权冗余**：MACD 研究已证明复权差异 < 2%，每个指标只保留 qfq 一种，立刻省 2/3 存储
2. **日线指标考虑直接查 stk_factor_pro**：不需要独立存 MACD/MA/RSI 日线表，需要时直接 JOIN stk_factor_pro
3. **只在周线/月线独立存储**：这些 Tushare 不提供，必须自算

## 八、最终结论

| 问题 | 回答 |
|------|------|
| 能不能统一？ | **能**，推荐分层方案（统一基础层 + 独立信号层） |
| 用什么复权？ | **指标用 qfq，收益用 hfq**，分开存 |
| Tushare 能用多少？ | 日线指标 **80% 可直接从 stk_factor_pro 取**，只需补 RSI14 和自定义 BIAS |
| 需要补什么？ | RSI14、自定义 BIAS、周/月线指标、信号检测逻辑、收益率回溯 |
