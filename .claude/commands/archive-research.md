# 归档已完成研究任务

> **处理 `task/` 下所有已完成的研究任务。**

## 沉淀去向

| 内容类型 | 去向 | 说明 |
|---------|------|------|
| 跨主题方法论/约束 | `base/` 或 `base/constraints/` | 新增或追加到已有文件 |
| 跨主题通用经验 | `base/跨主题通用经验.md` | 追加新验证的共性结论 |
| 主题基础设施（信号定义、Schema、数据源） | `{topic}/base/` | 该主题自己管理 |
| 已验证结论 | 确认已在 `{topic}/report/` | 没有则迁移 |
| 分层执行计划（Layer 步骤） | 丢弃 | 过程文档，已完成无价值 |
| 进度记录（已完成/待办清单） | 丢弃 | 过程文档 |
| 残存待办 | 新建 task 文件到 `task/{topic}/` | 仍有价值的未完成项 |

## 步骤

### 1. 扫描

遍历 `task/` 下所有主题文件夹中的 `.md` 文件。识别已完成的任务（状态标记 `DONE`/`已完成`/`✅`）。状态不明确的列出让用户确认。

### 2. 逐文件分析

对每个已完成的 task：

1. **先读目标文件夹已有内容**：读 `base/` 和 `{topic}/base/` 下所有现有文件
2. 识别跨主题内容（方法论/约束/通用经验）→ 已有相同主题的文件则整合追加，没有才新建
3. 识别主题基础设施（信号定义、Schema）→ 同上，整合到已有文件，不重复创建
4. 确认结论已沉淀到 `{topic}/report/`
5. 识别残存待办，判断是否需要新建 task

**整合原则**：沉淀前必须先读目标文件夹已有文件，相同主题的内容合并到已有文件中，不要重复创建新文件。

### 3. 确认

向用户展示每个 task 的沉淀方案：
- 哪些内容写到哪里
- 哪些确认已存在可丢弃
- 哪些待办需新建 task

确认后执行。

### 4. 写入 + 删除

执行沉淀，删除已归档的 task 文件。空的主题文件夹一并删除。

### 5. 汇报

输出结果：归档了哪些任务，沉淀了哪些文件，新建了哪些 task。
